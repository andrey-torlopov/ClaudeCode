# /refactor-plan - Планирование рефакторинга

<purpose>
Анализ участка Rust-кода, определение code smells и технического долга, создание пошагового плана рефакторинга с приоритетами и оценкой рисков.
</purpose>

## Когда использовать

- Планирование рефакторинга модуля/крейта
- Оценка технического долга
- Подготовка к миграции (sync -> async(tokio), callbacks -> async/await)
- Декомпозиция God-модулей

## Когда НЕ использовать

- Поиск багов (используй `/rust-review`)
- Разведка репо (используй `/repo-scout`)
- Простое переименование

## Входные данные

| Параметр | Обязательность | Описание |
|----------|:--------------:|----------|
| Scope | Обязательно | Путь к файлу, модулю или директории |
| Goal | Опционально | Конкретная цель рефакторинга |

**Tools first:** Анализируй молча. В чат - только путь к отчету.

## Алгоритм

### Шаг 1: Scope Analysis

1. Определи файлы для анализа
2. Подсчитай метрики:
   - Количество файлов и строк
   - Количество типов (struct/enum/trait/impl)
   - Зависимости между модулями (use/mod)
3. Прочитай CLAUDE.md для понимания архитектуры проекта

### Шаг 2: Code Smells Detection

Проверь каждый файл на:

**Structural smells:**
- God Module: mod.rs / lib.rs > 500 строк или > 10 публичных функций
- Long Function: функция > 50 строк
- Large File: файл > 500 строк
- Deep Nesting: > 4 уровня вложенности

**Design smells:**
- God-module (>500 строк в одном mod.rs)
- Fat Model (бизнес-логика в структурах данных)
- Тесная связность (concrete types вместо trait objects/generics)
- Нарушение Single Responsibility

**Rust-specific smells:**
- Callback hell вместо async/await
- `std::sync::Mutex` в async контексте (вместо `tokio::sync::Mutex`)
- `.unwrap()` / `.expect()` в production коде
- `Rc<RefCell<T>>` циклы без `Weak`
- `unsafe` блоки без обоснования
- Излишнее `.clone()` (проблемы с ownership)
- `String` вместо `&str` в параметрах функций
- `Box<dyn Error>` вместо typed errors (thiserror/anyhow)
- Mutable shared state без синхронизации

### Шаг 3: Dependency Mapping

1. Построй граф зависимостей между модулями/типами в scope
2. Найди циклические зависимости
3. Определи "узкие горлышки" (типы от которых зависит многое)

### Шаг 4: Refactoring Plan

Для каждого найденного smell создай шаг рефакторинга:

| Поле | Описание |
|------|----------|
| Приоритет | P0 (блокер) / P1 (высокий) / P2 (средний) / P3 (низкий) |
| Что | Конкретное описание проблемы |
| Действие | Конкретное описание решения |
| Файлы | Список затрагиваемых файлов |
| Риск | Низкий / Средний / Высокий |
| Зависимости | От каких шагов зависит |

**Приоритизация:**
- P0: Panics, data races (unsafe), memory leaks (Rc cycles)
- P1: Архитектурные нарушения, блокирующие развитие
- P2: Code smells, tech debt
- P3: Стилистика, мелкие улучшения

### Шаг 5: Verification Plan

Для каждого шага рефакторинга укажи способ проверки:
- `cargo build` - компиляция после изменений
- `cargo test` - прогон тестов
- `cargo clippy` - линтер
- Ручная проверка (если автоматизация невозможна)

### Шаг 6: Report Generation

Сохрани план в путь указанный пользователем или `audit/refactor-plan.md`.
