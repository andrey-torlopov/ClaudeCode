# /refactor-plan - Планирование рефакторинга инфраструктуры

<purpose>
Анализ инфраструктурного кода, определение config smells и технического долга, создание пошагового плана рефакторинга с приоритетами и оценкой рисков.
</purpose>

## Когда использовать

- Планирование рефакторинга IaC-модулей/конфигураций
- Оценка технического долга инфраструктуры
- Подготовка к миграции (Docker Compose -> Kubernetes, manual -> Terraform)
- Декомпозиция монолитных скриптов и конфигураций

## Когда НЕ использовать

- Ревью инфраструктуры (используй `/infra-review`)
- Разведка репо (используй `/repo-scout`)
- Простое переименование ресурсов

## Входные данные

| Параметр | Обязательность | Описание |
|----------|:--------------:|----------|
| Scope | Обязательно | Путь к файлу, модулю или директории |
| Goal | Опционально | Конкретная цель рефакторинга |

## Verbosity Protocol

**Tools first:** Анализируй молча. В чат - только путь к отчету.

## Алгоритм

### Шаг 1: Scope Analysis

1. Определи файлы для анализа
2. Подсчитай метрики:
   - Количество файлов и строк
   - Типы конфигураций (Dockerfile, .tf, .yml, .sh, Makefile)
   - Зависимости между файлами (source, include, import)
3. Прочитай `COMMON.md` (если есть) для понимания стека и конвенций проекта

### Шаг 2: Config Smells Detection

Проверь каждый файл на:

**Structural smells:**
- God-Makefile: Makefile > 300 строк или > 20 целей
- God-Script: Bash-скрипт > 200 строк без функций
- Fat Dockerfile: Dockerfile > 100 строк без multi-stage
- Deep Nesting: > 4 уровня вложенности в YAML/HCL
- Large File: любой конфиг > 500 строк

**Design smells:**
- Config drift: одинаковые параметры расходятся между окружениями
- Дублирование конфигураций между сервисами/модулями
- Тесная связность между компонентами (жестко прописанные адреса/порты)
- Нарушение DRY: copy-paste блоки в Terraform/Ansible

**DevOps-specific smells:**
- Mutable infrastructure: ручные изменения на серверах вместо IaC
- Manual steps в pipeline: TODO-комментарии с ручными шагами
- No rollback strategy: деплой без возможности отката
- No monitoring: сервисы без health checks и алертов
- Hardcoded values: IP-адреса, пароли, порты в коде
- No modules: повторяющийся Terraform-код без модулей
- Latest tags: Docker-образы без фиксированных версий
- No resource limits: контейнеры без limits/requests

### Шаг 3: Dependency Mapping

1. Построй граф зависимостей между компонентами в scope
2. Найди циклические зависимости
3. Определи "узкие горлышки" (компоненты от которых зависит многое)
4. Определи порядок деплоя (topological sort)

### Шаг 4: Refactoring Plan

Для каждого найденного smell создай шаг рефакторинга:

| Поле | Описание |
|------|----------|
| Приоритет | P0 (блокер) / P1 (высокий) / P2 (средний) / P3 (низкий) |
| Что | Конкретное описание проблемы |
| Действие | Конкретное описание решения |
| Файлы | Список затрагиваемых файлов |
| Риск | Низкий / Средний / Высокий |
| Зависимости | От каких шагов зависит |
| Валидация | Команда для проверки (terraform validate, shellcheck, и т.д.) |

**Приоритизация:**
- P0: Утечка секретов, сломанный пайплайн, отсутствие rollback
- P1: Архитектурные нарушения, блокирующие масштабирование
- P2: Config smells, tech debt, дублирование
- P3: Стилистика, мелкие улучшения, именование

### Шаг 5: Report Generation

Сохрани план в путь указанный пользователем или `audit/refactor-plan.md`.
